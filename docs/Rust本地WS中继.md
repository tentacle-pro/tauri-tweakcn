## Rust 本地 WS 中继
在这个方案中，TypeScript 前端不需要改写任何 WebSocket 逻辑，甚至可以直接抄 Google 官方文档里的代码，仅仅是换个 URL。

工作原理：

Rust 端 (中继器): 在 Tauri 启动时，Rust 悄悄在本地（例如 127.0.0.1:17891）启动一个非常轻量的 WebSocket Server。

TS 前端 (调用者): 前端代码依然写 const ws = new WebSocket("ws://127.0.0.1:17891");。对前端来说，它以为自己连的就是 Gemini。

代理转发 (核心): 当前端把音频 PCM 数据通过这个本地 WS 发给 Rust 时，Rust 收到数据，立刻通过它内部建立好的 “SOCKS5 通道（连着 gost）” 转发给真实的 Google Gemini WSS 接口。

优势：

前端代码零侵入： 前端的重连机制、数据包组装、音视频流处理，完全照搬 Google 的示例代码即可，不需要改成 Tauri 特有的 API。

安全闭环： 真正的 Gemini API Key 放在 Rust 里，在 Rust 转发请求给 Google 时动态拼接到 HTTP Header 或 URL 参数里，前端根本碰不到 Key。

因为 WebSocket 协议本身就自带了非常完善的连接状态控制帧（Close Frame）。我们只需要把 Rust 设计成一根“透明的纯水管”，不需要专门设计复杂的指令协议。

## 以下是三个问答式的方案解析和架构设计：

1. 通道异常断开，需要专门设计机制反馈给前端吗？
不需要专门设计，利用 WebSocket 协议的级联效应即可。

场景： 假设网络波动，或者 Gemini 服务器主动断开了与 Rust 的连接。

Rust 的处理： Rust 端的 WebSocket 客户端（连着 Gemini 的那一头）会收到一个 Close 事件或底层 TCP 错误。此时，你在 Rust 代码里只需做一件事：立刻主动关闭本地的 127.0.0.1:17891 连接。

前端的表现： 一旦 Rust 关了本地连接，前端 TypeScript 中你写的 websocket.onclose 或 websocket.onerror 回调函数会立刻触发。

结果： 对前端 TS 来说，它根本不知道中间有个 Rust 代理。它只会觉得“啊，我和 Gemini（其实是本地代理）断开了”，然后按照标准的 Web 开发逻辑去处理 UI 提示即可。

2. 重连机制是由 TypeScript 发起还是 Rust 自动处理？
强烈建议由前端 TypeScript 发起（前端驱动大脑，Rust 只做苦力）。

为什么不让 Rust 自动重连？ 如果 Rust 发现 Gemini 断了，自己偷偷重连，那前端 TS 传过来的音频流在这期间应该怎么处理？丢弃还是缓存？这会让 Rust 的状态管理变得极其复杂。

最佳实践（无状态中继）： 1. Rust 只要断连，就直接掐断前端的本地连接（如上述第 1 点）。
2. 前端 TS 监听到 onclose，在界面上显示“正在重新连接...”，然后执行重新 new WebSocket("ws://127.0.0.1:17891") 的逻辑。
3. Rust 监听到前端有了新的连接请求，就再去向 Gemini 发起一次全新的代理连接。

好处： 这样设计，前端完全可以复用 Google 官方示例或开源社区的断线重连代码。Rust 永远保持“无状态”（Stateless），连上就传数据，断了就销毁线程，代码极其干净、稳健。

3. 前端主动断开（如游戏结束）需要通知 Rust 吗？
不需要发特殊指令消息，前端直接调用 ws.close() 即可。

流程： 游戏结束时，你在 TypeScript 里执行 websocket.close()。

级联反应： 1. Rust 的本地 Server 会收到来自前端的 Close 帧（EOF）。
2. Rust 代码捕获到这个帧后，立刻调用内部客户端去关闭与 Gemini 的真实连接。
3. Rust 随后销毁这个中继任务，释放内存。

总结：“透明水管”架构的核心原则
为了不让系统变得复杂，请在开发时牢记这个设计原则：

前端 TS 是“大脑”： 负责生命周期（何时连、何时断、断了要不要重连）。

后端 Rust 是“盲目的水管工”： * 左边进水（收到前端数据），就往右边送（通过 SOCKS5 发给 Gemini）。

右边进水（收到 Gemini 回复），就往左边送（发给前端）。

任何一边的管子断了，立刻把另一边也掐断，然后收工下班。

这种设计能让你的桌面应用既拥有 Rust 底层代理的安全防护，又保留了 React/TS 生态里极其顺滑的 UI 状态管理体验。
